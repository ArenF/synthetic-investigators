# CoC 7e 역할/메커니즘 vs 현재 시스템 비교

> 분석일: 2026-04-22

---

## 1. 참여자 역할

### CoC 7e 원본

| 역할 | 책임 |
|------|------|
| **키퍼 (Keeper/GM)** | 장면 묘사, NPC 연기, 판정 시점/기술/난이도 결정, 결과 서술, 페이싱 |
| **탐사자 (Investigator)** | 행동 선언, 질문, 대화, 판정 수행, 전투 행동 선택 |

핵심: **키퍼가 "판정이 필요한 순간"을 판단**하는 것이 CoC의 근본 구조.
탐사자가 "문을 따겠다"고 선언하면, 키퍼가 "자물쇠따기 판정 필요"라고 결정한다.

### 현재 시스템

| 역할 | 담당 | 현재 구현 |
|------|------|----------|
| **키퍼 (GM)** | 인간 유저 | `send_turn`으로 장면 묘사, `judgment_request`로 판정, `npc_speak`로 NPC 대화 |
| **탐사자** | AI 모델 | `thinkingTakeTurn()`으로 응답 생성 (내면 → 행동) |

**현재 구조의 핵심 특징**: GM이 모든 것을 선제적으로 결정한다.
AI는 행동을 "선언"할 뿐, 그 행동이 판정이 필요한지 판단하는 것은 GM.

## 2. 게임 페이즈 (Phase) 비교

### CoC 7e의 게임 페이즈

```
┌─────────────────────────────────────────────────────────┐
│  1. 조사/탐색 (Investigation)                            │
│     자유 형식. 턴 순서 없음. 선언 → 판정 → 결과 반복     │
│     CoC 플레이 시간의 80%+를 차지                         │
├─────────────────────────────────────────────────────────┤
│  2. 사회적 상호작용 (Social Encounters)                   │
│     NPC 대화, 설득/위협/매혹. 대항 판정 가능               │
│     자유 형식이지만 대항 판정 규칙 있음                    │
├─────────────────────────────────────────────────────────┤
│  3. 전투 라운드 (Combat Rounds)                           │
│     DEX 순서. 라운드당 1행동. 구조화된 턴 시스템           │
│     공격/회피/반격/도주 선택                               │
├─────────────────────────────────────────────────────────┤
│  4. 추격전 (Chase Sequences)                              │
│     이동 포인트 + 장애물. 별도 규칙                       │
├─────────────────────────────────────────────────────────┤
│  5. 정신감정 (Sanity Events)                              │
│     특정 트리거 시 SAN 체크. 광기 발현 시 별도 처리        │
└─────────────────────────────────────────────────────────┘
```

### 현재 시스템의 게임 페이즈

```
┌─────────────────────────────────────────────────────────┐
│  단일 페이즈: "GM 입력 → AI 순차 응답" 루프              │
│                                                         │
│  GM send_turn ──→ AI[0] 응답 ──→ AI[1] 응답 ──→ ...    │
│       ↑                                          │      │
│       └──────────── turn_complete ───────────────┘      │
│                                                         │
│  판정은 GM이 별도로 judgment_request 전송                │
│  결과는 pendingDiceResults에 저장 → 다음 send_turn에 주입│
└─────────────────────────────────────────────────────────┘
```

### 갭 분석

| CoC 페이즈 | 현재 시스템 | 상태 |
|-----------|-----------|------|
| 조사/탐색 | 단일 루프로 대체. GM이 장면 묘사 → AI가 행동 선언 | **부분 동작** — 기본 흐름은 되지만 아래 문제들 |
| 사회적 상호작용 | NPC 대화는 chatLog에만 존재 (AI 히스토리 미주입) | **불완전** |
| 전투 라운드 | 없음. 턴 순서는 수동 설정이지만 DEX 자동 정렬 없음 | **미구현** |
| 추격전 | 없음 | **미구현** |
| 정신감정 | SAN 감소 시 자동 광기 판정은 있음 | **부분 동작** — 광기 증상 테이블 없음 |

## 3. 키퍼(GM)가 수행하는 행동 — 상세 비교

### 3.1 장면 전환 / 묘사

| CoC 7e | 현재 시스템 | 상태 |
|--------|-----------|------|
| 장면 묘사 (서술) | `send_turn` gmText | **동작** |
| 특정 탐사자에게만 정보 전달 | `targetIds`로 지정 가능 | **동작** |
| 전체 장면 전환 | `send_turn` targetIds 전체 | **동작** |
| 시간 경과 묘사 | 텍스트로만 가능 (시스템적 시간 없음) | **텍스트만** |

| CoC 7e | 현재 시스템 | 상태 |
|--------|-----------|------|

| 판정 결과 → 서술 | `naturalLanguage` 생성 + `pendingDiceResults`로 다음 턴에 주입 | **동작** |
| 난이도 설정 (보통/어려움/극한) | `difficulty: regular / hard / extreme` | **동작** |
| **밀어붙이기 (Pushing)** | — | **미구현** |
| **대항 판정 (Opposed Roll)** | — | **미구현** |
| **보너스/페널티 다이스** | — | **미구현** |
| **행운 소비로 판정 보정** | — | **미구현** |
| **결합 판정 (Combined Roll)** | — | **미구현** |

### 3.3 NPC 운영

| CoC 7e | 현재 시스템 | 상태 |
|--------|-----------|------|
| NPC 대사 전달 | `npc_speak` → chatLog + 클라이언트 | **절반** — AI 히스토리에 미주입 |
| NPC 소개 (인지) | `introduce_npc` → SessionState.knownNpcs | **동작** |
| NPC가 특정 탐사자에게만 말함 | `introduce_npc`에 `targetIds` 있음 | **부분** — npc_speak에는 targetIds 없음 |
| NPC 감정/태도 변화 | — | **미구현** |
| NPC 전투 행동 | — | **미구현** |

### 3.4 상태 관리

| CoC 7e | 현재 시스템 | 상태 |
|--------|-----------|------|
| HP 증감 | `adjust_stat` (hp) + 판정 Effect | **동작** |
| SAN 증감 + 자동 광기 | `adjust_stat` (san) + `applySanLoss()` 자동 광기 | **동작** |
| MP 증감 | `adjust_stat` (mp) 서버 지원 | **서버만** — UI 없음 |
| 행운 증감 | `adjust_stat` (luck) 서버 지원 | **서버만** — UI 없음 |
| 아이템 획득/분실 | 판정 Effect (`item_gain`/`item_lose`) | **동작** |
| 기술값 변경 (성장) | 판정 Effect (`skill`) 정의됨 | **미구현** — 정의만 있고 적용 안 됨 |
| 부상 상태 | `injuries[]` 배열 있음 | **텍스트만** — 메커니즘적 효과 없음 |

## 4. 탐사자(AI)의 행동 유형 — 상세 비교

### CoC 7e에서 탐사자가 할 수 있는 모든 것

```
조사/탐색 페이즈
├── 장소 탐색: "방을 둘러본다" → GM이 흔적발견/청취 판정 호출
├── 물건 조사: "서류를 읽는다" → GM이 도서관이용/역사 등 판정 호출
├── 이동: "2층으로 올라간다" → GM이 장면 전환
├── NPC 대화
│   ├── 정보 요청 → GM이 NPC 반응 + 필요시 설득/위협 판정
│   ├── 설득/위협/매혹 → 대항 판정 (탐사자 vs NPC의 POW/심리학)
│   └── 거짓말 → 심리학 대항 판정
├── 지식 활용: "이 문양을 알아본다" → 오컬트/크툴루신화 판정
└── 준비 행동: 아이템 사용, 장비 교체 등

전투 라운드
├── 공격: 격투/사격 판정 → 피해 굴림
├── 회피: 회피 판정 (피공격 시)
├── 반격: 격투 판정 (회피 대신, 성공 시 반격 피해)
├── 도주: 이탈 시도 → 추격전 전환 가능
├── 밀어붙이기: 실패한 판정 재시도 (실패 시 더 큰 대가)
├── 행운 소비: 판정 결과 보정
└── 전투 기동: 기습, 엄폐, 원거리 사격 등

정신감정 이벤트
├── SAN 체크: 실패 시 SAN 손실
├── 일시적 광기: 5+ 손실 시 자동 발동 → 광기 증상 테이블
├── 무기한 광기: 누적 1/5 초과 → 기저 광기
├── 정신분석: 타인의 광기 치료 시도
└── 공포증/집착 발동
```

### 현재 시스템에서 AI가 할 수 있는 것

```
AI 응답 (thinkingTakeTurn)
├── Stage 1: 내면/OOC 서술 (감정, 생각, 전략)
└── Stage 2: 행동 묘사 (말, 동작, 반응)
    └── 텍스트 출력이 전부. 시스템적 행동 선언 메커니즘 없음
```

### 핵심 차이

| 측면 | CoC 7e 원본 | 현재 시스템 |
|------|-----------|-----------|
| **행동 선언** | 탐사자가 구체적으로 선언 → GM이 판정 여부 결정 | AI가 서술형 텍스트 출력. **구조화된 행동 선언 없음** |
| **판정 타이밍** | 행동 선언 → 판정 → 결과 → 서술 (실시간 루프) | GM이 **별도로** judgment_request → 결과를 **다음 턴에** 주입 |
| **행동-판정 연결** | 하나의 흐름 | **분리됨** — 행동과 판정 사이에 시간차 |
| **선택지** | 복수의 기계적 옵션 (공격/회피/반격/도주) | 자유 서술만 |

## 6. 문제점 정리 — 턴 시스템 관점

### 6.1 행동-판정 분리 문제

**CoC 원본**: "문을 따겠다" → (즉시) 자물쇠따기 판정 → 결과 → "문이 열렸다/안 열렸다"
**현재 시스템**: AI가 "문을 따겠다"고 행동 묘사 → (GM이 읽고) 판정 모달 열어서 judgment_request → (결과 나옴) → GM이 다음 send_turn으로 결과 전달 → AI가 이제야 결과를 알고 반응

**결과**: 한 번의 자연스러운 행동에 **2~3턴이 소요**됨.

```
CoC 원본:  선언 → 판정 → 결과 (1 사이클)
현재:      선언(턴1) → GM이 판정 실행 → 결과 전달(턴2) → AI 반응(턴3)
```

### 6.2 동시 행동 vs 순차 처리

전체 대상으로 `send_turn`을 보내면 `for (const charId of targetIds)` 루프로 순차 처리.
AI[0]의 행동이 AI[1]의 컨텍스트에 `accumulatedContext`로 주입됨.

- **조사 페이즈**: 각자 다른 곳을 탐색하는 상황에서도 순서가 생김
- **전투 페이즈**: DEX 순서 없이 turnOrder 수동 설정만 가능

### 6.3 AI의 행동이 비구조적

AI가 출력하는 것은 자유 서술 텍스트뿐이다.
"문을 딴다", "경찰에 전화한다", "괴물을 공격한다"가 **모두 같은 형태의 텍스트**.

GM이 이 텍스트를 읽고 → 판정이 필요한지 판단 → 수동으로 판정 모달을 여는 과정이 필요.
이건 CoC 원본과 유사하지만, **GM의 부담이 AI 수만큼 배로 증가**하는 문제.

### 6.4 판정 결과의 지연 전달

`pendingDiceResults`는 다음 `send_turn` 때 GM 메시지 앞에 텍스트로 붙여진다.
→ GM이 `send_turn`을 보내지 않으면, AI는 판정 결과를 **영원히 모름**
→ CLAUDE.md에 "판정 후 자동 AI 턴 실행" 이슈로 기록되어 있음

### 6.5 NPC 대화의 단절

```
현재: GM → npc_speak → chatLog (클라이언트에만 표시)
                         └→ AI 히스토리에는 들어가지 않음

필요: GM → npc_speak → chatLog + AI 히스토리에 주입
                       → 다음 AI 턴에서 NPC 대화를 인지
```

### 6.6 밀어붙이기 (Pushing) 부재

CoC 7e의 상징적 메커니즘: 판정 실패 시 "밀어붙이기"로 재시도 가능, 대신 실패하면 더 심각한 결과.
현재 시스템에는 "실패 후 재시도" 경로가 전혀 없음.

### 6.7 행운 소비 부재

CoC 7e: 판정 직후 행운을 소비해서 roll을 보정할 수 있음 (예: roll 55, target 50 → 행운 5 소비로 성공).
현재: 행운은 숫자로만 존재. 판정에 개입하는 메커니즘 없음.

## 7. 개선 방향 — 세 가지 접근법

### 접근법 A: 현재 구조 유지 + 편의성 개선

가장 보수적. 현재의 "GM 주도, 자유 서술" 구조를 유지하되 마찰을 줄임.

| 개선 | 설명 |
|------|------|
| 판정 후 자동 AI 턴 | judgment_result 후 자동으로 send_turn 실행 (옵션) |
| NPC 대화 → AI 주입 | npc_speak 시 AI 히스토리에도 넣기 |
| GM 입력 textarea | 멀티라인 장면 묘사 |
| 빠른 판정 버튼 | AI 응답에서 "이 행동은 판정 필요" → 원클릭 판정 |

**장점**: 최소 변경. 현재 코드 구조 유지.
**단점**: 행동-판정 분리 문제는 그대로. GM 부담 여전히 높음.

### 접근법 B: 행동 선언 구조화

AI가 자유 서술 + **구조화된 행동 선언**을 함께 출력.
GM이 선언을 승인/수정 → 필요시 자동 판정.

```
AI 출력:
{
  "inner": "이 문 뒤에 뭔가 있을 것 같다...",
  "action": "조심스럽게 문고리를 잡고 돌린다",
  "intent": {
    "type": "skill_use",
    "skill": "자물쇠따기",
    "target": "잠긴 문"
  }
}

→ GM에게 "자물쇠따기 판정 진행?" 프롬프트 표시
→ GM 승인 시 자동 판정 + 결과 즉시 AI에게 전달
→ AI가 결과를 반영한 후속 행동 출력
```

**장점**: 행동-판정이 하나의 사이클로. GM 부담 감소.
**단점**: AI 출력 파싱이 복잡해짐. 프롬프트 엔지니어링 부담 증가.

### 접근법 C: 페이즈 기반 시스템

게임 상태에 현재 페이즈를 명시적으로 관리.

```
session.phase: 'exploration' | 'social' | 'combat' | 'chase' | 'sanity_event'

exploration: 현재와 유사 (자유 행동)
combat: DEX 순서 강제, 행동 선택지 제한 (공격/회피/반격/도주)
social: 대항 판정 지원
sanity_event: 광기 테이블 + 증상 자동 적용
```

**장점**: CoC 규칙에 가장 충실. 전투가 체계적.
**단점**: 구현 복잡도 높음. 현재 코드 대폭 변경 필요.

## 8. 권장 순서

현실적으로 **A → B 순서**가 좋다고 판단.

```
Phase 1 (A): 마찰 제거                     ← 지금 바로 할 수 있음
├── 판정 후 자동 AI 턴 옵션
├── NPC 대화 → AI 히스토리 주입
├── GM 입력 textarea
└── MP/행운 UI 추가

Phase 2 (B): 행동 선언 구조화               ← 프롬프트 실험 후
├── AI 출력에 intent 필드 추가
├── GM 승인 → 자동 판정 파이프라인
├── 밀어붙이기 (Pushing) 구현
└── 행운 소비 메커니즘

Phase 3 (C): 페이즈 시스템 (선택)           ← 전투가 중요해지면
├── 전투 라운드 (DEX 순서)
├── 대항 판정
├── 추격전
└── 광기 증상 테이블
```

Phase 1만으로도 GM 운영 경험이 크게 개선되고,
Phase 2는 이 프로젝트의 "AI 행동 실험" 비전에 가장 큰 데이터를 줄 수 있다 (AI가 구조화된 의사결정을 하게 되므로).
