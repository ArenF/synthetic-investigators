Critical CR-01 — Combined Roll: OR 조건에서 실패 roll이 "최악의 성공"으로 선택됨
server/game/judgment.ts:275-279

Combined roll에서 overallSuccess가 true(OR 조건, 1개 이상 성공)일 때, outcome을 결정하는 reduce가 전체 rolls를 순회한다. 실패 roll은 rank가 0이므로 어떤 성공 roll(rank 1~3)보다 "나쁜 것"으로 선택되어, 성공으로 판정됐는데 outcome이 regular_failure나 fumble이 될 수 있다. 

// 현재 코드: overallSuccess인데도 failure tier가 선택될 수 있음
const outcome: JudgmentOutcomeKey = overallSuccess && rolls.length > 0
  ? (rolls.reduce((worst, r) => {
      const rank = { extreme_success: 3, hard_success: 2, regular_success: 1 }
      return (rank[r.outcome] ?? 0) < (rank[worst.outcome] ?? 0) ? r : worst
    }, rolls[0]).outcome as JudgmentOutcomeKey)
  : 'regular_failure'

수정 제안: OR 조건에서는 successes만 순회해야 한다.

// AND면 rolls 전체(전부 성공), OR면 successes만 순회
const pool = request.condition === 'and' ? rolls : successes
const outcome: JudgmentOutcomeKey = overallSuccess && pool.length > 0
  ? (pool.reduce((worst, r) => {
      const rank = { extreme_success: 3, hard_success: 2, regular_success: 1 }
      return (rank[r.outcome] ?? 0) < (rank[worst.outcome] ?? 0) ? r : worst
    }, pool[0]).outcome as JudgmentOutcomeKey)
  : 'regular_failure'
Critical CR-02 — applyEffects 'skill' kind: 불변이어야 할 CoCCharacter를 직접 변이

Warning CR-04 — Combined Roll: 스킬별로 다른 fumble threshold 적용
server/game/judgment.ts:236

Combined roll에서 단일 d100을 굴리지만, 각 스킬별로 fumbleThreshold를 개별 계산한다 (baseSkill < 50 ? 100 : 96). 같은 주사위 결과에 대해 한 스킬은 fumble이고 다른 스킬은 bad_failure일 수 있다.

CoC 7e 규칙상 combined roll의 fumble 판정은 주 스킬(primary skill) 기준으로 통일하는 것이 자연스럽다. 

const rolls: RollResult[] = request.skills.map(skill => {
  const baseSkill = char.skills[skill] ?? 0
  const fumbleThreshold = baseSkill < 50 ? 100 : 96  // 스킬마다 다름
  ...
})

수정 제안: primary skill의 fumble threshold를 모든 스킬에 통일 적용.

Warning CR-05 — Opposed Roll: resolveAccept가 sideA만 처리
server/game/judgment.ts:396-401

resolveAccept()는 pending.rolls[0] (sideA)에 대해서만 effect를 적용한다. Opposed roll에서 sideB(패배 측)에 대한 outcomes/effects가 정의되어 있어도 적용되지 않는다.

현재 opposed roll의 outcomes는 승자(sideA) 관점으로만 설계되어 있으므로 즉각적인 버그는 아니지만, "패배 시 sideB에게 피해" 같은 effect를 설정할 수 없는 구조적 제한이다.

Warning CR-06 — sanMax가 san.starting을 사용 (san.max와 불일치)
server/api.ts:187 | client/src/components/CharacterStatus.tsx:97

getCharacterStates()에서 sanMax: char.derived.san.starting을 사용한다. CoC 7e에서 SAN의 최대값은 99 - 크툴루 신화(%)이며, 초기값은 POW이다. 이 두 값은 다르다.

Design CR-08 — applySanLoss와 performSanCheck의 광기 판정 중복
server/game/state.ts:59-78 | server/game/judgment.ts:592-607

광기 판정 로직이 두 곳에 분산되어 있다:

  • GameState.applySanLoss(): 누적 SAN 손실 기반 무기한 광기 자동 판정
  • performSanCheck(): 5점 이상 손실 시 INT 굴림 기반 일시적 광기 판정

applySanLoss()는 일시적 광기를 체크하지 않고, performSanCheck()는 무기한 광기를 직접 체크하지 않는다(applySanLoss에 위임). 이 분리 자체는 의도적이지만, adjust_stat으로 GM이 수동으로 SAN을 깎을 때 applySanLoss()를 거치므로 무기한 광기는 자동 트리거되지만 일시적 광기(INT 굴림)는 트리거되지 않는다.

GM 수동 조정 시에는 의도적일 수 있으나, 이 동작 차이를 인지해야 한다.

Note CR-21 — ChatMessage ID 충돌 가능성
server/api.ts:398 | client/src/App.tsx:68

서버와 클라이언트 모두 Date.now() 기반으로 ChatMessage ID를 생성한다. 같은 밀리초에 여러 메시지가 생성되면 ID가 충돌할 수 있다. 서버 측: `${Date.now()}-${charId}`로 charId를 붙여 완화하고 있으나, 클라이언트 측 thinking placeholder: `thinking-${charId}-${Date.now()}`는 같은 charId에 대해 밀리초 충돌 시 중복된다.

현재 규모에서는 실질적 문제가 아니지만, crypto.randomUUID()나 카운터 기반 ID가 더 안전하다.
