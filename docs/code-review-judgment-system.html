1-1. server/game/dice.ts

심각 S-01: d100WithBonusPenalty — 연산자 우선순위 버그 dice.ts:119

chosenTens + unitsDie === 0은 JS 연산자 우선순위에 의해 chosenTens + (unitsDie === 0 ? ...)으로 평가된다. unitsDie=0일 때 삼항연산자가 100을 반환하여 chosenTens + 100이 되며 — 100을 초과하는 값(예: 110)이 발생한다.

// 현재 (버그)
const roll = chosenTens + unitsDie === 0 ? 100 : chosenTens + unitsDie

// 수정
const sum = chosenTens + unitsDie
const roll = sum === 0 ? 100 : sum
영향: d100 주사위 결과가 100을 초과할 수 있어 모든 기술 판정 비교가 깨진다. 페널티 다이스가 가장 큰 영향을 받는다 (높은 십의 자리 + 일의 자리 0).

1-2. server/game/judgment.ts

심각 S-02: rollCombinedCheck — reduce() 초기값 누락 judgment.ts:272-277

rolls.reduce((worst, r) => ...)에 초기 누산값이 없다. 초기값 없는 reduce는 첫 번째 요소를 누산값으로 사용하고 인덱스 1부터 반복을 시작한다. 즉, rolls[0]은 비교 대상에서 제외되며 — 그대로 기본값으로 통과된다.

// 현재
rolls.reduce((worst, r) => {
  return (rank[r.outcome] ?? 0) < (rank[worst.outcome] ?? 0) ? r : worst
})

// 수정 — 명시적 초기값
rolls.reduce((worst, r) => {
  return (rank[r.outcome] ?? 0) < (rank[worst.outcome] ?? 0) ? r : worst
}, rolls[0])
영향: 복합 판정에서 잘못된 결과 등급이 보고될 수 있다. rolls[0]이 최악의 결과를 가지고 있으면 제대로 비교되지 않는다.
심각 S-03: rollGroupCheck — 동일한 reduce() 버그 judgment.ts:339-342

동일한 패턴: successes.reduce((best, r) => ...)에 초기값이 없다. 성공이 1개뿐이면 콜백이 실행되지 않아 — 우연히 정확한 결과가 반환된다 (그 하나의 요소). 하지만 2개 이상이면 첫 번째가 비교에서 누락된다.

수정: reduce()의 두 번째 인자로 , successes[0]을 추가.

1-4. server/api.ts

심각 S-11: judgment_resolve — 이중 해결 경쟁 조건 api.ts:1048-1106

두 개의 연결된 클라이언트가 동시에 judgment_resolve를 보낼 수 있다. 둘 다 sess.pendingJudgment를 non-null로 읽고, 둘 다 resolveJudgment()를 호출하고, 둘 다 효과를 적용한다. 판정이 두 번 해결되어 스탯 변동이 두 배가 된다.

// 현재 (취약)
if (!sess.pendingJudgment) { ... break }
const final = resolveJudgment(sess, sess.pendingJudgment, resolution)
sess.pendingJudgment = null

// 수정 — 획득 후 즉시 클리어
if (!sess.pendingJudgment) { ... break }
const pending = sess.pendingJudgment
sess.pendingJudgment = null  // 비동기 작업 전에 즉시 클리어
try {
  const final = resolveJudgment(sess, pending, resolution)
  ...
} catch (err) {
  sess.pendingJudgment = pending  // 에러 시 복원
  ...
}
영향: 효과 이중 적용 — HP/SAN/행운이 두 배로 차감. 멀티 클라이언트 세션에서 치명적 데이터 손상.
