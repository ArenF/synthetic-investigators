아키텍처 개요

Express + WebSocket 서버로, GM(게임 마스터)이 텍스트를 보내면 각 캐릭터를 담당하는 AI 모델이 순서대로 응답하는 구조다. 서버는 단일 프로세스이며 모든 세션은 인메모리 Map으로 관리된다.

타입 시스템

핵심 타입 계층

// CoCCharacter — 캐릭터 JSON 파일에서 로드, 세션 중 불변
interface CoCCharacter {
  id: string                           // 파일명과 동일 e.g. "jisu"
  characteristics: CharacterCharacteristics  // STR/CON/SIZ/DEX/APP/INT/POW/EDU
  derived: DerivedStats                 // hp/mp/san/luck/build/moveRate/damageBonus
  skills: SkillSet                      // 인덱스 시그니처 포함 — 조회 시 ?? 0 필수
  backstory: Backstory                  // 10개 배경 필드
  equipment: Equipment                  // items(string[]) / weapons / cash
  modelConfig: ModelConfig              // provider / model / temperature
  sessionState?: SessionState           // 옵셔널 — 런타임에는 GameState가 따로 관리
}

// SessionState — GameState.states Map에서 캐릭터별로 관리되는 뮤터블 상태
interface SessionState {
  hp: number;  san: number;  mp: number;  luck: number
  temporaryInsanity: boolean;  indefiniteInsanity: boolean
  injuries: string[]
  currentItems: ItemObject[]            // 세션 중 추가/제거 가능
  sessionSanLoss: number               // 누적 SAN 손실 (광기 기준치 판정용)
  knownNpcs: KnownNpc[]
}

// Effect — 판정 결과의 Side Effect 유니온
type Effect =
  | { kind: 'stat';      stat: 'hp'|'san'|'mp'|'luck'; delta: number }
  | { kind: 'skill';     skill: string; delta: number; permanent?: boolean }  // ⚠️ 미구현
  | { kind: 'item_gain'; item: ItemObject }
  | { kind: 'item_lose'; itemName: string }
  | { kind: 'status';    status: 'temporaryInsanity'|'indefiniteInsanity'; value: boolean }

// TurnContext — AI에게 매 턴 전달되는 입력
interface TurnContext {
  character: CoCCharacter;  sessionState: SessionState
  scenarioId: string;       turnNumber: number
  gmMessage: string          // 컨텍스트 요약 + 주사위 결과 + GM 텍스트 + 이전 AI 응답 합산
  visibleHistory: TurnRecord[]  // 최근 8턴
  playMode: PlayMode;       coCharacters: CoCharacter[]
}

두 가지 상태 저장소

CoCCharacter (불변)

캐릭터 JSON 파일에서 로드. 기본 능력치/기술값/배경 저장. 세션 중 수정 안 됨.

GameState (뮤터블)

세션 중 HP/SAN 변화, 아이템 추가·제거, 광기 상태 추적. states: Map<charId, SessionState> 구조.

createSession(sessionId, setup) {
  // ① 캐릭터 JSON 로드 → 불변 Map
  for (charId of setup.characterIds) {
    const char = loadCharacter(charId)     // characters/{id}.json
    characters.set(char.id, char)
    players.set(char.id, createPlayer(char, mode))  // factory → Claude/Gemini/OpenAI/Ollama
    state.addCharacter(char)               // GameState 초기화 (HP/SAN 등)
  }

  // ② ScenarioManager — 기존 파일 있으면 로드, 없으면 새 세션
  scenario = new ScenarioManager(sessionId, ...)

  // ③ GameSession 객체 구성 후 sessions Map에 등록
  sessions.set(sessionId, session)
}

runTurn() 상세 (캐릭터 순서 루프)

async function runTurn(session, gmText, targetLabel, targetIds, previousResponses) {

  // ① 판정 결과 드레인 — pendingDiceResults에 쌓인 결과를 한 번에 꺼냄
  const pendingDice = session.pendingDiceResults.splice(0)
  const diceContext = pendingDice.map(r => `[판정 결과 — ...]`).join('\n')

  // ② 캐릭터별 순차 실행
  for (const charId of targetIds) {

    // ③ 히스토리 압축 — 전체 턴 로드 후 최근 8턴 full + 이전 요약
    const allTurns = scenario.getRecentTurns(1000)           // ⚠️ 매 캐릭터마다 호출
    const charTurns = allTurns.filter(t => t.characterId === charId)
    const contextMessages = buildContextMessages(charTurns)   // 8턴 full + 이전 1줄 요약

    // ④ gmMessage 조립: [요약] + [주사위결과] + [GM텍스트] + [이번 턴 다른 AI 행동]
    const gmWithContext = summary ? `${summary}\n\n${accumulatedContext}` : accumulatedContext

    // ⑤ TurnContext 빌드 → AI 턴 실행
    const record = await player.thinkingTakeTurn(ctx)

    // ⑥ statsBefore/After 보정 (스냅샷 기반)
    record.statsBefore = statsBefore.get(charId)
    record.statsAfter  = state.getState(charId)   // 현재 상태

    // ⑦ attempt 감지 → attempt_declared broadcast
    if (record.response.attempt) {
      broadcast(session, { type: 'attempt_declared', ... })
    }

    // ⑧ 다음 캐릭터를 위해 누적 컨텍스트 업데이트
    previousResponses.push({ charName, text: record.response.action })
    accumulatedContext = buildTurnContext(gmText, diceContext, previousResponses)

    // ⑨ 저장 — scenario.addTurn() + logger.record()
  }

  session.turnNumber++
  broadcast(session, { type: 'state_update', characters: getCharacterStates(session) })
}

판정 시스템

skillCheck() — 6-tier 계산 (dice.ts)

roll = d100()   // 1~100 랜덤

// 난이도에 따른 목표값 조정
target = skillValue                      // regular
target = Math.floor(skillValue / 2)      // hard
target = Math.floor(skillValue / 5)      // extreme

fumbleThreshold = skillValue < 50 ? 100 : 96  // CoC 7e 규칙
regularFailureMax = Math.floor((skillValue + 95) / 2)

// 판정 우선순위 (위에서부터 체크)
roll >= fumbleThreshold  → 'fumble'
roll <= target/5         → 'extreme_success'
roll <= target/2         → 'hard_success'
roll <= target           → 'regular_success'
roll <= regularFailureMax → 'regular_failure'
else                     → 'bad_failure'

발견된 문제들

Bug 순환 의존성 — judgment.ts → api.ts

server/game/judgment.ts:13에서 import type { GameSession } from '../api.js'. 게임 레이어가 API 레이어에 의존하는 역전된 구조.

개선안: performJudgment() 시그니처를 (state: GameState, characters: Map<string, CoCCharacter>, charId, ...)로 변경하여 게임 레이어만 참조하도록 분리.

Bug ClaudePlayer — historySnapshot 선언 후 미사용

claude-player.ts:40: const historySnapshot = this.history.length 선언 후 splice(historySnapshot)가 없다. API 실패 시 폴백으로 super.thinkingTakeTurn()을 호출하는데, 그 전에 history를 건드리지 않으므로 실제 버그는 아니지만 dead code.

Dead Field pendingStatsBefore — GameSession 필드에 설정만 하고 읽지 않음

api.ts:298: session.pendingStatsBefore = statsBefore로 저장하지만 이 필드를 읽는 코드가 없다. statsBefore는 runTurn() 지역 변수로 이미 충분히 추적된다.

개선안: GameSession에서 pendingStatsBefore 필드 제거.

Performance getRecentTurns(1000) — 매 캐릭터 턴마다 전체 히스토리 로드

api.ts:317: N명의 캐릭터가 있으면 한 턴에 N번 전체 히스토리를 꺼낸 뒤 filter(). 세션이 길어질수록 매 턴 비용 증가.

개선안: ScenarioManager.getCharacterHistory(charId, limit)는 이미 있음. getRecentTurns(1000) 대신 getCharacterHistory(charId, 50) 사용.

Design dice_roll vs judgment_request — Effect 적용 불일치

구버전 dice_roll은 판정 결과를 HP/SAN에 반영하지 않는다. judgment_request는 applyEffects()를 통해 자동 적용. GM이 어떤 경로를 쓰느냐에 따라 동일한 판정도 게임 상태에 다른 결과를 낸다.

개선안: dice_roll 핸들러 제거 또는 내부적으로 performJudgment()를 호출하도록 위임.

Design skill Effect — applyEffects()에서 silently 무시

state.ts:150: Effect 유니온에 skill kind가 있고 JudgmentOutcomes에서 기술 성장 효과를 정의할 수 있지만, applyEffects()에 case 'skill'이 없어 조용히 무시된다. 에러도, 로그도 없음.

Design introduce_npc — 에러 묵살

api.ts:1000: try { ... } catch {} — 에러가 발생해도 클라이언트나 로그에 아무것도 남지 않음.

Design 세션 메모리 누수 — isProcessing 중 클라이언트 전원 종료

api.ts:1090: 마지막 클라이언트가 끊길 때 isProcessing === true면 sessions.delete()가 실행되지 않아 세션이 메모리에 영구 잔류. AI 호출이 진행 중인 상태에서 모든 클라이언트가 연결을 끊고 재연결하지 않으면 GC 불가.

Design GeminiPlayer — 주석과 코드 불일치

gemini-player.ts:23: // Fix: include ALL messages in history (not slice(0, -1))라고 쓰여있지만 실제 코드는 messages.slice(0, -1)을 사용 중. 주석이 의도와 반대로 기록됨.

Type buildTurnMessage — ItemObject 방어적 타입 체크

prompt-generator.ts:241: typeof i === 'string' ? ... : i — currentItems는 타입상 ItemObject[]인데 런타임에 string이 들어올 수 있음을 방어. GameState.addCharacter()에서 string을 ItemObject로 변환하므로 이 방어 코드가 실제로 필요한 경로가 있는지 확인 필요.

Info ClaudePlayer — (this.client as any) 캐스팅

claude-player.ts:53: Extended Thinking API 호출 시 SDK 타입 지원 여부 확인 후 캐스팅 제거 가능. 현재 SDK 버전에서 betas 파라미터 및 thinking 옵션이 타입에 포함되어 있는지 점검 권장.

Info KNOWN_MODELS — 하드코딩된 모델 목록

api.ts:663: Claude/Gemini/OpenAI 모델 목록이 하드코딩되어 있어 새 모델 출시 시 수동 업데이트 필요. Ollama만 실제 API 조회. 허용 가능한 설계이나 유지보수 부담 존재.
